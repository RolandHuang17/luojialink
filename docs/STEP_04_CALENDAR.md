# Step 04：日历时间片、共同空闲与阶段留痕

更新时间：2026-06-13

## 1. 本轮开发前已经实现了什么

截至 Step 04 开始前，项目已经具备比较完整的本地 MVP 主链路：

- 可以使用 `alice`、`bob`、`carol` 进行 mock 登录。
- 可以查看广场、发布请求、查看请求详情。
- 可以提交“想搭”申请。
- 发起者可以确认或拒绝申请。
- 确认后系统会创建临时会话。
- 会话页可以查看会话列表、读取消息、发送文本消息。
- 已有 `docs/STEP_02_APPLICATION_SESSION.md` 和 `docs/STEP_03_CHAT_MESSAGES.md` 记录前两阶段开发。

当时的核心缺口是：用户没有维护空闲时间的入口，会话里也无法展示双方共同空闲。

## 2. 本轮开发实现了什么

本轮目标是补上演示中最能体现“智能匹配”的日历能力。

### 2.1 后端新增能力

新增日历接口：

- `GET /api/calendar/slots`：获取当前用户时间片。
- `PUT /api/calendar/slots`：覆盖保存当前用户时间片。
- `GET /api/calendar/common-free?sessionId=...`：获取会话双方共同空闲。
- `GET /api/calendar/conflicts?postId=...`：检查当前用户忙碌时间与请求时间是否冲突。

业务规则：

- 继续使用已有 `CalendarSlot` 表，不新增数据库表。
- 时间片只支持 `available` 和 `busy` 两种状态。
- 保存时间片时校验日期、开始时间、结束时间和状态。
- 结束时间必须晚于开始时间。
- 共同空闲由双方 `available` 时间片交集计算，并限制在请求的 `startTime/endTime` 范围内。
- 冲突提示只检查当前用户 `busy` 时间片和请求时间是否重叠。
- 共同空闲接口只允许会话双方访问。

### 2.2 小程序新增能力

日历页：

- 展示我的时间片列表。
- 支持通过表单选择日期、开始时间、结束时间和状态。
- 支持新增本地时间片。
- 支持删除时间片。
- 支持保存到后端。
- 保存后刷新后端返回的数据。

会话页：

- 选择会话时同步加载共同空闲。
- 在消息区上方展示“共同空闲”卡片。
- 如果暂无共同空闲，提示用户可以先在聊天里确认时间。

### 2.3 测试覆盖

新增后端集成测试覆盖：

- 用户可以保存并读取自己的时间片。
- 非法时间片参数会返回 400。
- Alice/Bob 匹配后，双方 available 时间片有交集时能返回共同空闲。
- 双方没有交集时返回空数组。
- 非会话成员不能查询该会话共同空闲。
- busy 时间片与请求时间重叠时能返回冲突提示。

## 3. 当前可演示流程

现在可以演示：

1. Alice 和 Bob 分别进入“日历”页，保存有重叠的空闲时间片。
2. Alice 发布请求。
3. Bob 点击“想搭”。
4. Alice 确认申请。
5. 双方进入“会话”页。
6. 选择新会话后，可以看到共同空闲时间。
7. 双方继续通过消息确认具体安排。

这意味着项目已经具备“发布、申请、确认、会话、发消息、共同空闲”的核心演示闭环。

## 4. 后续规划

下一轮建议优先做：

- AI/模板建议：根据活动类型、地点、共同空闲和标签生成破冰建议。
- POI 地点：查询本地 mock 地点并用于推荐。
- 举报页完善：从小程序提交真实举报。
- 演示脚本：编写 `docs/DEMO_SCRIPT.md`，固定 10 分钟验收路径。
- 测试清单：编写 `docs/TEST_CASES.md`，验收前逐项自测。
- UI 打磨：补空状态、错误态和关键操作提示。

## 5. 本轮不做的内容

- 不做拖拽式日历。
- 不做复杂重复日程。
- 不做微信订阅提醒。
- 不阻止冲突请求发布，只做提示。
- 不接真实地图或地点服务。
- 不处理微信开发者工具自动生成的本地配置文件。
