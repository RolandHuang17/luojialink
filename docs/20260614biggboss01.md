## 3. 正确启动方式

新版项目不要再优先使用：

```bash
npm run db:init
```

原因：`server/prisma/init.sql` 仍可能滞后于新版 `schema.prisma`，会建出旧表结构，导致类似下面的错误：

```text
The column `avatarUrl` does not exist in the current database.
```

推荐后端启动/重置流程：

```bash
cd D:\Projects\luojialink\server
npm install
copy .env.example .env
npx prisma db push
npm run db:seed
npm run dev
```

验证：

```bash
npm run build
npm test
```

当前已验证结果：

```text
npm run build 通过
npm test 通过：5 个测试文件，8 个测试用例全部通过
```

健康检查地址：

```text
http://127.0.0.1:3000/api/health
```

## 4. 已实现功能对照

### 4.1 注册 onboarding

已实现：

- 学院、年级、年龄、昵称、头像、性别、家乡、微信号。
- 常驻地点。
- MBTI 四个引导问题。
- 期待关系：搭子 / 朋友。
- 一句话介绍自己。
- 爱好。
- 最喜欢的三件事。
- 给对方的一句话。
- 绝对禁忌 / 雷点。
- 个人特质，必须选择 3 个。
- 多数字段为锁死选项，用户不能随意输入。
- 每屏只展示一个问题。
- 完成后展示欢迎页并进入广场。

待改进：

- 头像目前是 URL 输入，体验生硬。
- 基础信息和自我介绍信息的分段感不明显。
- 横向流式动画还不明显。
- 视觉仍偏工程化，需要重做排版和动效。

### 4.2 广场

已实现：

- 底部五入口：广场、日程、+、消息、我的。
- 广场顶部四分类：吃饭、运动、自习、娱乐。
- 卡片流展示帖子。
- 卡片包含头像、昵称、性别、年级、学院、标题、时间、地点等信息。
- 点击卡片进入详情。
- 点击发布者可进入个人主页。
- 详情页可点击“想搭”提交申请。

待改进：

- 卡片 UI 需要重做，当前更像普通信息列表。
- 信息层级不够突出，一眼看到的重点还不够明确。
- 默认只看“吃饭”，是否需要增加“全部”待讨论。
- 分类色彩和风格需要更强区分。

### 4.3 发布和草稿箱

已实现：

- 发布标题、具体规划、活动地点、活动时间、发布板块。
- 发布按钮。
- 保存草稿。
- 草稿箱在“我的”中可进入。
- 草稿可继续编辑或删除。

待改进：

- 发布页仍偏普通表单，缺少“创建活动计划”的产品感。
- 费用偏好字段在旧模型中仍存在，但新版发布页未显式填写。
- 草稿字段校验可能偏严格，需要手测。

### 4.4 消息和申请

已实现：

- 消息页统一展示 pending 申请和正式会话。
- 别人申请我的想搭，会在消息页出现窗口。
- 我申请别人的想搭，也会在消息页出现窗口。
- pending 状态不能自由聊天。
- 发布者可接受 / 拒绝申请。
- 申请者可撤回申请。
- 接受后创建正式会话。
- 正式会话可以发送文本消息。
- 双方成功搭上后可以互换联系方式。
- 微信号默认隐藏，双方互换后才显示。

待改进：

- 消息页视觉不像微信式 inbox，仍需重排。
- pending 申请窗口和正式聊天窗口的状态差异需要更明显。
- 互换联系方式的流程提示需手测并优化。

### 4.5 我的

- 申请管理和消息页存在功能重叠，后续需统一产品入口。
- 页面视觉像按钮集合，缺少个人中心层级。

### 4.6 日程

已实现：

- 近 14 天横向日期视图。
- 红色表示我发布的活动。
- 蓝色表示我申请的活动。
- 绿色表示已经搭上的活动。
- 点击活动进入详情。
- 后端接口 `/api/calendar/events` 聚合日程。
- 后端在发布、申请、接受申请时检查时间冲突。

待改进：

- 视觉不是苹果日历，更像横向日期 + 当日事件列表。
- 没有月视图。
- 无法直接在日程页编辑时间。
- 早期 available/busy 时间片功能与 Step 06 事件日历之间需要统一说明。

## 5. 主要问题判断

当前 UI 的主要问题不是单纯字体或动画，而是：

1. 版面排版和信息层级落后。
2. 多数页面仍像工程表单或功能列表。
3. 视觉风格不够统一。
4. 关键页面缺少产品感。
5. 动画是锦上添花，应在排版和层级之后处理。

优先重做页面建议：

1. 广场：第一印象最重要。
2. onboarding：注册流决定产品质感。
3. 消息页：需要清晰区分申请和正式聊天。
4. 发布页：从普通表单改成创建计划。
5. 我的页：从按钮集合改成个人中心。

  - 但建议先重做布局和信息层级，再做动效。
- AI 破冰还没有真正接入远程 AI：
  - 目前是模板建议。
  - 后续如果要真实 AI，需要接 API Key、补异常兜底，并注意不要把真实姓名、学号、联系方式传给模型。
- 头像还没有实现本地上传：
  - 当前仍偏测试态。
  - 后续应接微信头像能力或本地上传/选择能力。
- 登录仍是测试账号逻辑：
  - 新注册用户会留在本地数据库里。
  - 测试阶段可以用 `npm run db:seed` 重置。
  - 真正上线应按微信 openid 自动登录。
  - 如果不上线，也可以保留账号密码登录/注册作为演示方案。
- 红点目前不是严格后端未读系统：
  - 本地缓存能支持演示，但跨设备、清缓存、复杂多账号切换时仍可能不准确。
  - 正式方案应在后端增加 `readAt` 或 `lastReadMessageId`。
- 仍可能存在未发现 bug：
  - 必须用微信开发者工具完整跑 A/B/C 多账号流程。
  - 尤其要测注册、发布、申请、同意、拒绝、撤回、聊天、互换联系方式、日程冲突、草稿箱、编辑资料。

### 建议下一位同学的优先级（仅供参考，按自己的想法来）

1. 先完整手测主链路，把阻塞流程的 bug 全部列出来。
2. 优先修会影响演示的 bug：注册失败、发布失败、申请状态错乱、聊天进不去、联系方式提前暴露、日程冲突漏判。
3. 重做广场卡片和详情页，因为这是用户进入后的第一印象。
4. 重做 onboarding 的文案、布局和过渡，让注册像“了解你”而不是“填表”。
5. 重做消息页状态表达，清楚区分“申请中”和“正式会话”。
6. 重做发布页，让它像“创建一场校园计划”，而不是普通表单。
7. 再统一 Logo、色彩、空状态、加载状态和动画。
8. 最后再考虑真实 AI、真实微信登录、头像上传、后端未读系统等更重的能力。
